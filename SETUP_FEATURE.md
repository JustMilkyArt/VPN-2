# Доработка: Автоматическая настройка сервера при создании

## Общая концепция

После нажатия **"Добавить"** в форме создания сервера:
1. Сервер сохраняется в БД со статусом `настройка...`
2. Форма закрывается
3. Открывается **модальное окно прогресса** поверх панели
4. Пошагово выполняется настройка с живым логом
5. По завершении — статус меняется на `онлайн` или `не настроен`

---

## Статусы сервера

| Статус | Когда |
|--------|-------|
| `настройка...` | В процессе установки |
| `онлайн` | Все шаги пройдены успешно |
| `не настроен` | Хотя бы один шаг завершился ошибкой |
| `офлайн` | Был онлайн, стал недоступен (существующая логика) |

---

## Модальное окно прогресса — UX

- Каждый шаг — строка: иконка состояния + название + детали
- Иконки: `⏳` в процессе → `✅` успех / `❌` ошибка / `⚠️` предупреждение
- У шагов с подшагами — раскрывающийся список подшагов
- При ошибке на любом шаге: лог останавливается, показывается причина, две кнопки — **"Повторить шаг"** и **"Отменить создание"**
- "Отменить создание" удаляет сервер из БД
- После успешного завершения — кнопка **"Готово"**, окно закрывается, сервер появляется в списке

**Критические ошибки** (блокируют продолжение, сервер не добавляется):
- Шаг 1: SSH недоступен / неверные credentials

**Некритические ошибки** (можно повторить или пропустить, сервер добавляется со статусом `не настроен`):
- Любой другой шаг

## Шаги настройки

### Шаг 1 — Проверка подключения к серверу
**Одинаково для EU и RU**

Подшаги:
- SSH-коннект на исходный порт 22
- Выполнение `echo OK`, проверка ответа
- Определение прав пользователя (root или sudo)

Результат успеха: соединение установлено, права подтверждены

---

### Шаг 2 — Настройка безопасности
**Для EU и RU — одинаково, но с разными подшагами по SSH**

#### 2.1 Обновление системы
- `apt update && apt upgrade -y`
- Установка `unattended-upgrades` для автоматических патчей безопасности

#### 2.2 Настройка Fail2Ban
- Установка `fail2ban`
- Конфиг: блокировка после 5 неудачных попыток, бан на 1 час

#### 2.3 Настройка UFW
- Установка и включение `ufw`
- Разрешённые порты:
  - **EU**: SSH (временно 22), 443 (VLESS+NaiveProxy), 51820/UDP (AWG)
  - **RU**: SSH (временно 22), 443, 51820/UDP, 2408/UDP (WARP)
- Всё остальное — закрыть

#### 2.4 Генерация нового SSH-ключа *(EU и RU)*
- Генерируем новую пару Ed25519 на admin-сервере
- Добавляем публичный ключ в `~/.ssh/authorized_keys` на целевом сервере
- Проверяем подключение по новому ключу (не удаляя старый)
- После успешной проверки — удаляем старый ключ из `authorized_keys`
- Приватный ключ — шифруем и сохраняем в БД

#### 2.5 Смена пароля root
- Генерируем случайный пароль (24 символа, буквы+цифры+спецсимволы)
- Устанавливаем через `chpasswd`
- Сохраняем зашифрованно в БД

#### 2.6 Отключение password-аутентификации SSH
- **EU**: `PasswordAuthentication no` в `/etc/ssh/sshd_config`, restart sshd
- **RU**: проверяем текущий статус — если уже отключено, фиксируем; если включено — отключаем

#### 2.7 Смена SSH-порта
- Генерируем случайный порт из диапазона 10000–65000
- Прописываем в `sshd_config`
- Обновляем UFW: закрываем 22, открываем новый порт
- Restart sshd
- Проверяем подключение на новый порт
- Сохраняем новый порт в БД

### Шаг 3 — Установка стека

#### EU сервер

| Подшаг | Что делается |
|--------|-------------|
| 3.1 Xray-core | Установка, генерация Reality-ключей (x25519), базовый конфиг |
| 3.2 AmneziaWG | Установка AWG + awg-tools, генерация серверных ключей |
| 3.3 NaiveProxy + Caddy | Установка бинарника NaiveProxy и Caddy. Если в системе есть свободный поддомен типа `naiveproxy_eu` со статусом `active` → привязываем, Caddy настраивается с доменом и SSL. Если нет → ⚠️ домен не привязан |

**WARP для EU — не устанавливается.**

#### RU сервер

| Подшаг | Что делается |
|--------|-------------|
| 3.1 Xray-core | Установка, конфиг для приёма пользователей + исходящий туннель на EU (если EU-сервер есть в системе — линкуем автоматически, если нет — конфиг-заготовка) |
| 3.2 AmneziaWG | Установка AWG + awg-tools |
| 3.3 NaiveProxy + Caddy | Аналогично EU, но ищем поддомен типа `naiveproxy_ru` |
| 3.4 WARP | Установка `warp-cli`, регистрация, настройка режима routing, включение автостарта |

### Шаг 4 — Сбор информации о сервере

Собирается и записывается в БД, отображается в разделе **Параметры**:

| Параметр | Видимость в панели |
|----------|--------------------|
| SSH-порт (новый) | Всем |
| SSH-пользователь | Всем |
| SSH-пароль | Creator / Гл.админ — под TOTP; Админ — скрыто (звёздочки) |
| SSH-ключ (приватный) | Creator / Гл.админ — под TOTP; Админ — скрыто (звёздочки) |
| Часовой пояс сервера | Всем |
| Публичный ключ Xray (Reality publicKey) | Всем |
| AWG публичный ключ сервера | Всем |
| Версии сервисов (Xray, Caddy, AWG, WARP если RU) | Всем |

---

### Шаг 5 — Финальная проверка

| Проверка | Метод | Результат при неудаче |
|----------|-------|-----------------------|
| SSH-доступ по новому порту и ключу | SSH-коннект | ❌ Ошибка |
| Xray запущен | `systemctl is-active xray` | ❌ Ошибка |
| AmneziaWG запущен | `systemctl is-active awg-quick@wg0` + `awg show` | ❌ Ошибка |
| NaiveProxy/Caddy запущен | `systemctl is-active caddy` | ❌ Ошибка |
| Caddy принимает HTTPS (если домен привязан) | HTTPS-запрос на поддомен | ⚠️ Предупреждение |
| WARP запущен (только RU) | `warp-cli status` | ❌ Ошибка |
| Fail2Ban активен | `systemctl is-active fail2ban` | ⚠️ Предупреждение |
| UFW активен и правила корректны | `ufw status verbose` | ⚠️ Предупреждение |
| Пинг сервера | ICMP | ⚠️ Предупреждение |

- ❌ **Ошибка** — шаг помечается failed, предлагается "Повторить шаг"
- ⚠️ **Предупреждение** — фиксируется в логе, не блокирует завершение

## Дополнительная кнопка в Параметрах сервера

**"Сменить SSH-пользователя"** — отдельная кнопка в разделе Параметры:
- Поле ввода нового имени пользователя
- После подтверждения — создаётся новый пользователь на сервере, переносятся ключи, обновляется БД
- Доступна только при статусе сервера `онлайн`
- Только для ролей: Creator и Главный админ

## Изменения в БД

Новые поля в таблице `servers`:

| Поле | Тип | Описание |
|------|-----|----------|
| `setup_status` | string | `pending / in_progress / done / failed` |
| `ssh_private_key` | text (encrypted) | Сгенерированный приватный ключ |
| `ssh_password_encrypted` | text (encrypted) | Зашифрованный пароль root |
| `ssh_port_new` | int | Новый SSH-порт после смены |
| `xray_public_key` | text | Reality publicKey |
| `awg_server_public_key` | text | AWG публичный ключ |
| `server_timezone` | string | Часовой пояс |
| `xray_version` | string | Версия Xray |
| `caddy_version` | string | Версия Caddy |
| `awg_version` | string | Версия AWG |
| `warp_version` | string | Версия WARP (только RU) |
| `naiveproxy_subdomain_id` | int (FK) | Привязанный поддомен для NaiveProxy |

## Новые бэкенд-эндпоинты

| Метод | URL | Описание |
|-------|-----|----------|
| `POST` | `/servers/{id}/setup` | Запустить настройку сервера |
| `GET` | `/servers/{id}/setup/status` | Текущий статус (polling каждые 2 сек) |
| `POST` | `/servers/{id}/setup/retry-step` | Повторить конкретный шаг |
| `DELETE` | `/servers/{id}/setup/cancel` | Отменить настройку и удалить сервер |
| `POST` | `/servers/{id}/change-ssh-user` | Сменить SSH-пользователя |

---

## Что не входит в эту доработку

- Создание VPN-подключений (VLESS, AWG, NaiveProxy) — отдельная операция
- Настройка домена/поддомена с нуля (только автопривязка при наличии готового)
- Смена SSH-пользователя при создании — только кнопка в Параметрах после создания
- Полная реализация TOTP-подтверждения — место заложено, логика ролей уже есть
